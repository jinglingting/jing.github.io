MySQL碎片整理

Posted on 2019-03-02 | In MySQL优化

存储引擎为MyISAM, 定期碎片整理

  • 对表存储引擎为MyISAM进行整理, 使用 optimize table table_name;
  • 对表存储引擎为InnoDB进行整理, 使用 alter table table_name engine=innodb;
    • 对于存储引擎为InnoDb的表整理, 也可以使用 optimize table table_name, 只是会映射为alter table ...的命令
    • MySQL手册中提到

      For InnoDB tables, OPTIMIZE TABLE is mapped to ALTER TABLE … FORCE, which rebuilds the table
      to update index statistics and free unused space in the clustered index. This is displayed in the output of
      For InnoDB tables, OPTIMIZE TABLE is mapped to ALTER TABLE … FORCE, which rebuilds the table
      to update index statistics and free unused space in the clustered index. This is displayed in the output of
      OPTIMIZE TABLE when you run it on an InnoDB table, as shown here:

1.创建表,然后添加数据,确定表存储的物理文件的大小以及表的信息

  • 创建数据表,添加数据,然后复制数据使表中数据有一定规模

  • 定位mysql存储文件的位置

  • 查看此时mysql存储文件的大小

  • 通过information_schema数据库查看mysql_optimization数据库的test100表的信息:

2.执行删除数据操作, 再次确认表的物理文件大小以及表的信息

  • 然后执行delete操作, 删除id%3=1的数据,因为都是复制,能删除掉1/3的数据

  • 再次查看mysql存储文件的大小

  • 再次通过information_schema数据库查看test100表的信息

3.对表进行optimize操作,再次查看表的物理文件大小以及表的信息

  • 数据存储文件的大小没有变化,对于这种情况,应该对使用MyISAM存储引擎的表定期整理.

  • 然后再次查看mysql存储文件的大小

  • 第三次通过information_schema查看test100表的信息

4.分析:

  1. 在使用delete后:
    • test100表的数据文件没有发生变化;
    • 通过information_schema数据库的TABLES表查看test100的信息发现TABLE_ROWS,DATA_LENGTH,DATA_FREE字段发生变化.
  2. 在使用optimize 整理test100表后:
    • test100表的数据文件大小发生了变化,减小了进1/3
    • information_schema的TABLES表中的关于test100表的信息发生变化.data_free项为0了.而且之前的data_length - data_free = 此次的data_length

5.命令补充:

  • 直接使用命令 show table status like 'test100'\G即可查看test100的信息,效果等同于在information_schema数据库通过TABLES查看
